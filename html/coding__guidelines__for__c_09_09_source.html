6 The following guidelines are based on a draft document from Baudouin Raoult,
7 and were updated following the OOPS code review 11-15 July 2011.

9 \section files_extensions Files Extensions
10 - Each class is defined in two files:
11  - The header file contains the class definition and has the extension
12  <tt>.h</tt>.
13  - The implementation file defines the methods and has the extension
14  <tt>.cc</tt>
15 - The file names should match the name of the class.
16 - <tt>.cc</tt> and <tt>.h</tt> files must be in the same directory.
17 
18 \section header_files Header Files
19 - Header files should follow the layout suggested in
20  \link ExampleHeader Documents/ExampleHeader.h\endlink.
21 - The <tt>public</tt>, <tt>protected</tt> and <tt>private</tt> sections
22  of a class should be declared in that order (i.e. <tt>public</tt> first,
23  <tt>private</tt> last).
24 - Use <tt>\#include</tt> guards to protect against multiple inclusions.
25 - The guard name should be constructed from the path to the <tt>.h</tt>
26  file, as it appears in a <tt>\#include</tt> statement (see below), but
27  with lowercase characters converted to uppercase, special characters
28  replaced by underscores, and with a trailing underscore.
29 - The <tt>\#endif</tt> for the guard should be followed on the same line
30  by // and then the name of the guard.
31 - All headers should be self-sufficient. A header should compile by itself.
32 - There should be one main class per file. Helper classes are allowed, as
33  long as they are only used from within the file.

35 \section code_documentation Code Documentation
36 - Documentation is generated using doxygen.
37  - Instructions on building the documentation are in the
38  \link README README\endlink file.
39  - The doxygen manual can be found
40  <A HREF="http://www.stack.nl/~dimitri/doxygen/manual.html">here</A>.
41  - See <A HREF="http://www.stack.nl/~dimitri/doxygen/formulas.html">here</A>
42  for instructions on including \f$\mbox{\LaTeX}\f$ mathematical formulae.
43 - Doxygen recognises special comments:
44  - A brief comment is a single line like this:
45 \verbatim /// This one-line comment gives a brief description of a class or method.\endverbatim
46  - A detailed comment looks like this:
47 \verbatim /*!
48  * This is a detailed comment. It has more than one line,
49  * and provides more complete information about a class or method.
50  */ \endverbatim
51 - Class definitions should be preceded by a doxygen brief comment
52  followed by a detailed comment.
53 - Method and member declarations should be preceded by a doxygen brief
54  comment, unless the role of the method or member is completely obvious from
55  its name. A detailed comment may also be provided.
56 - Normal C++ comments should be used in <tt>.cc</tt> files where necessary to
57  explain the internal logic of a function.
58 - The code should be self-explanatory. But, add comments to explain complex
59  algorithms.
60 - Do not comment-out code. Use the source-code management system!
61 - Don't include comments to indicate authorship or modification history.
62  That is what <tt>git blame</tt> is for!
63 - HTML links to auxiliary documents (e.g. pdf files) can be made to appear on
64  the \link Overview Overview\endlink page by adding them to
65  Documents/overview.h

67 \section includes Includes
68 - Do not include unnecessary headers. Use class forwarding. The compiler only
69  needs to see a class definition when calling methods or establishing the size
70  of an object. When referring to a pointer or a reference, the compiler does not
71  need to know the detail of the class.
72 - <tt>\#include<iosfwd></tt> instead of <tt>\#include<iostream></tt>,
73  if possible.
74 - Included ".h" files must use the full path from the build directory ("oops")
75 - Order of header files:
76  - Own ".h" file.
77  - C System headers (e.g. <tt><unistd.h></tt>).
78  - C++ system headers (e.g. <tt><iostream></tt>).
79  - Other library headers <tt><boost/...></tt>).
80  - Oops/util/logger project headers.
81 - Within each of these categories,use alphabetical ordering.

83 \section identifiers Identifiers
84 - Identifiers should be in "camel case". That is, they should be mainly
85  in lower case, with an upper case letter at the start of each internal
86  word: e.g. changeResolution.
87 - Do not use underscores to separate words.
88 - Class names should start with an upper case letter.
89 - Method and member names should start with a lower case letter.
90 - Member names have an underscore at the end.
91 - Use short identifiers for local variables, loop indices, etc.
92 - Use longer, meaningful names for methods, members, classes, etc.

94 \section use_of_const Use of Const
95 - Use const wherever possible.
96 - Avoid passing objects by const value. Pass by const reference instead.
97 - Remember. The rule when reading definitions is to work from right
98  to left. So, for example, <tt>char * const test</tt> means that
99  <tt>test</tt> is a <tt>const</tt> pointer to a <tt>char</tt> object:
100  <tt>
101  - const char * test = "xyz"; // non-const pointer to const data
102  - char * const test = "xyz"; // const pointer to non-const data
103  - const char * const test = "xyz"; // const pointer to const data
104  - bool isEmpty() const; // the isEmpty method does not change its object.
105  </tt>

113 - Make virtual functions private.

128 \section constructors_and_destructors Constructors and Destructors
129 - Always declare copy and assignment constructors. Make them private unless
130 you need to copy.
131 - If the default copy constructor is sufficient, include a comment to this
132  effect in the class definition.
133 - Write the constructors and the destructors at the same time.
134 - If the class has a virtual table, its destructor must be virtual.
135 - All resources allocated by an object must be deallocated in the destructor.
136 - Beware of partially constructed objects.
137 - Except for the copy constructor, single-argument constructors should be
138  declared <tt>explicit</tt> to prohibit implicit type conversions.
139 - Base class destructors must be either <tt>public</tt> and <tt>virtual</tt>,
140  or <tt>protected</tt> and not <tt>virtual</tt>.
141 - The copy constructor should copy all the data. However, you may wish to give it a second,
142 default argument to allow this behaviour to be over-ridden.
143 - Do not code <tt>MyClass a = b;</tt>
144  - This looks like assignment, but in fact calls the copy constructor.
145  - Code <tt>MyClass a(b);</tt> instead.

147 \section members Member Variables
148 - Member variables should always be private.
149 - Use accessors if you need to access a member variable from outside a class.
150 - Don't use the accessors from within the class. Use the member itself.
151 
152 \section accessors Accessors
153 - Accessors (a.k.a. getters and setters) should only be implemented if
154  necessary. They break the encapsulation.
155 - Accessors should be inline.
156 - Accessors must have the same name as the member (but without the underscore)
157  for example, the accessors for a member <tt>Foo foo_;</tt> should be:
158  - const Foo& foo() const {return foo_; }
159  - void foo(const Foo& f) { foo_ = f; }
160 
161 \section Methods
162 - A method is a request to an object to do something or to provide something.
163  The name of the method should reflect this.
164  - E.g. <tt>changeResolution</tt> is preferable to <tt>resolutionChanger</tt>.
165 - If a method is virtual in a base class, declare it as virtual in all derived
166  classes that override it.
167 
168 \section opoverload Operator Overloading
169 - Don't do it unless it is meaningful.
170 - Don't subvert the mathematical properties (associativity, etc.).
171 - Don't use an operator for conversion. Implement an "asDouble" method rather
172  than "operator double()".
173 
174 \section pointers Pointers and References
175 - Prefer references to pointers. If an object is guaranteed to exist, use a
176  reference.
177 - Passing or returning a non-const pointer means passing ownership of the
178  pointed object.
179 - Passing or returning a const pointer means keeping ownership of the pointed
180  object, and that the pointed object can be null
181 - In any other case, pass a reference to the object. Use const whenever the
182  object will not be modified.

184 \section static Use of Static
185  - Avoid <tt>static</tt> if possible.
186  - Be aware there are different types of static (function-local, file-scope).
187  - Be aware that static variables cause problems in multi-threaded applications.
188 
189 \section useofcasts Use of Casts
190  - Use c++ style casts.
191  - Avoid downcasting. It is a symptom of bad inheritance, or not enough
192  functionality in the base class
193  - Write "double(expression)", not "(double) expression"
194  - <i>Guideline for the use of const cast to be written...</i>.

196 \section c_code C Code
197 - Don't use C functions (e.g. <tt>printf</tt>) if C++ provides the same
198  functionality.
199 - If you must use a C function, prefix it with a double colon
200  (e.g. <tt>"::sleep(10)"</tt>)
201 - When possible, wrap any C function in a C++ object (e.g. Sleeper)
202 - Never use C style casts.
203 - For unsigned value, use a typedef: <tt>typedef unsigned long ulong;</tt>

205 \section preprocessor Preprocessor
206 - The preprocessor should only be used to define <tt>\#include</tt> guards in
207 <tt>.h</tt> files and for variables specified via the <tt>-D</tt> flag at
208 compile time.
209 - The preprocessor should not be used to define macros or constants.
210  - The only permitted macros are <tt>ABORT</tt>, <tt>ASSERT</tt> and
211  <tt>LOG</tt>, and macros defined in the <tt>boost</tt> library
212  (e.g. <tt>BOOST_AUTO_TEST_CASE</tt>).
213 - Don't pepper the code with ifdef's for machine/compiler dependent
214  conditional compilation. Put any such code in a header file that can be
215  included wherever needed.

217 \section namespaces Namespaces
218 - Model-independent code should be defined in the <tt>oops</tt> namespace.
219 - Model-specific code should be defined in a separate, model-specific
220  namespace.
221 - Do not <tt>use</tt> an entire namespace (i.e. using directive).
222 - By preference, use explicit namespace qualifications
223  (e.g. <tt>std::string</tt>).
224  However, <tt>using std::string</tt> etc. is acceptable.
225 - <tt>using</tt> statements must never be used at global scope in a header file.
226 - Use anonymous namespaces to restrict classes (e.g. Factories) to file scope.

228 \section readability Readability
229 - As far as possible, adhere to the rules listed in the
230  <A HREF="http://google-styleguide.googlecode.com/svn/trunk/cppguide.xml">
231  Google C++ Style Guide</A>. Note however that, contrary to the Google rules:
232  - We do use streams.
233  - We do (currently) use dynamic-casts
234  - We allow non-const references as arguments.
235 - Use <A HREF="http://google-styleguide.googlecode.com/svn/trunk/cpplint/">
236  cpplint</A> to check your code. You may wish to turn off the following
237  cpplint filters:
238  - <tt>build/include_alpha</tt> (Because our idea of alphabetic order is
239  different from Google's.)
240  - <tt>build/include_order</tt> (Because cpplint wrongly thinks boost header
241  files are c-system header files.)
242  - <tt>readability/streams</tt> (Because we use streams.)
243  - <tt>runtime/rtti</tt> (Because we use dynamic_casts.)
244 - Keep lines below 80 characters.
245 - Tab characters are not allowed.
246 - Indent class and function bodies, <tt>if</tt> and <tt>for</tt> blocks, etc.
247  - <tt>public</tt>,
248  <tt>private</tt> and <tt>protected</tt> labels in a class definition
249  should be indented one space with respect to the start of the class
250  definition.
251  - Use a two space indent for everything else.
252  - It is preferable to indent code inside a namespace block. (However, we
253  have many examples where this is not done.)
254 - Split long lines in a way that makes it obvious that the code continues
255  on the next line.
256 - Continuation lines should be indented.
257 - If you split an argument list, align the arguments with those on
258  the previous line.
259 - The opening brace should appear
260  on the same line as the argument list, initialisation list,
261  loop expression. etc.
262 - The closing brace should appear on its own line, and aligned with
263  the start of the statement it closes.
264 - Braces should be used for all control structures (<tt>if</tt>, <tt>for</tt>,
265  <tt>switch</tt>, etc.), even for "one-liners".
266 - The <tt>else</tt> statement should be on the same line as the closing
267  brace of the preceding block, and the opening brace of the following block.
268 - Don't declare more than one member per line.
269 - Don't initialise on the same line as you declare: (e.g. <tt>int i=3;</tt>).
270 - Only one statement per line.
271 - Remove whitespace at the end of a line.
272 - Add a space after a comma in an argument list
273 - All operators, except <tt>"!"</tt> should be surrounded by spaces.
274 - Separate inline comments from code by at least two spaces
275 - There should be a space after <tt>//</tt> (or after <tt>///</tt> in the
276  case of a Doxygen inline comment).
277 - If the initialisation list in a class definition is too long to be on the
278  same line, put in on the next line with the colon indented by 4 spaces.

280 \section optimization Optimization
281 - Pass objects by reference, not by value.
282 - Prefer initialization over assignment.
283 - Use <tt>++i</tt>, not <tt>i++</tt> when incrementing iterators.
284 - Use the initialization list to initialize member objects.

286 \section logging Logging
287 - All logging messages should use the <tt>Logger</tt> class.
288  Do not write to <tt>cout</tt> or <tt>cerr</tt>.
289 - The logger adds a newline at the end of each message, so you don't need to.
290 - <tt>endl</tt> forces an unnecessary buffer flush. Use <tt>\\n</tt> instead.
291 - Use the appropriate logging category:
292  - <b>Info</b> is for normal output
293  - <b>Trace</b> is for more verbose output that could help the user
294  understand the logical flow of the program.
295  - <b>Warning</b> is for non-fatal error messages.
296  - <b>Error</b> is for fatal error messages.
297  - <b>Debug</b> is for debugging. Code in the shared repository should
298  not output to this category.
299 
300 \section abort_and_assert ABORT, ASSERT and Error Handling
301 - Errors should abort
302 - Use the <tt>ABORT</tt> macro to exit after an error.
303  Do not call <tt>exit</tt> directly.
304 - Use the <tt>ASSERT</tt> macro liberally. It compiles to nothing unless
305  the <tt>CHECK_ASSERTS</tt> macro variable is set, and it helps the
306  reader to understand the code.
307 - Remember, asserts can be disabled. Your code should not change behaviour
308  if you disable the asserts. Use "if" and ABORT if you want something to be
309  always checked.
310 - Do not use exceptions (try/catch/throw)
311 
312 \section pointers Pointers and Smart Pointers
313 - Use references instead of pointers as much as possible.
314 - Use smart pointers in preference to c pointers.
315  - Use <tt>boost::scoped_ptr</tt> if possible,
316  otherwise <tt>boost::shared_ptr</tt>.
317 - Do not use <tt>auto_ptr</tt>.
