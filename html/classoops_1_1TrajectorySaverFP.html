Detailed Description

template<typename MODEL>
class oops::TrajectorySaverFP< MODEL >

Manages computation of linearisation trajectory

The TrajectorySaver manages the computation of the linearisation trajectory for the LinearModel and other terms of the cost function.

The linearisation trajectory is computed in two steps:

  1. The TrajectorySaver, itself a post-processor (inheriting from PostBase class), is enrolled in the CostFunction::linearize method to the PostProcessor class object (containing a vector of PostBase processors). The PostProcessor object is subsequently attached to the non-linear trajectory. During the trajectory it records the inner loop-resolution states which are forced from the non-linear model trajectory (at each LinearModel time step).
  2. Once the non-linear trajectory is finished, Jb, Jo and Jc terms of the cost function call their initializeTraj methods to set up post-processors that are enrolled into the TrajectorySaver and which shall be attached to the computation of the linearization trajectory of the LinearModel (e.g. CostJo::initializeTraj sets up an Observer that is tasked to compute the linearization trajectory of the LinearObsOperator) in the recordTrajectory method. The linearization trajectory is computed by the low resolution non-linear Model (set up to be consistent with the LinearModel). At each LinearModel time step, the State is forced (a set of user defined fields) using the recorded States (forced from the high resolution trajectory in step 1).

The computation of the linearization trajectory is split into two parts for practical reasons. While it would be possible to do the entire computation during the non-linear trajectory, it would be subsequently challenging to extract the LinearObsOpertor (set up with screened ObservatoinSpace for the inner loop) linearization state (GeoVaLs) from the full linearization state including all the observations (in case when screening is performed in the trajectory).
